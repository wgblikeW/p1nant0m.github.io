### 背景与动机

想象一下：你是一家 AI 公司的安全工程师，某天突然发现有人通过在 prompts 里玩"文字游戏"，让你的 AI 助手乖乖听话地泄露了内部机密... 这不是科幻，而是 **Prompt Injection（提示词注入）** 正在真实上演的安全问题。

随着大语言模型（LLM）在企业级应用中的广泛部署，Prompt Injection 攻击已成为不容忽视的安全威胁。所谓 Prompt Injection，是指攻击者通过精心构造的输入，试图诱导模型忽略原有指令、执行恶意操作或泄露敏感信息。

面对这一挑战，我们不禁要问：**能否训练一个专有领域的小模型来自动检测此类攻击？** 本文将详细介绍如何使用通义千问 Qwen3 的 0.6B、4B 和 8B 三个版本，构建一个专门用于 Prompt Injection 检测的分类模型。

#### 本文摘要

本文将从以下几个方面展开：
- 数据集构建：良性样本与恶意样本的采集与处理
- 模型训练：Qwen3 的微调方法与训练技巧
- 推理部署：基于 vLLM 的高性能推理方案
- 性能优化：分类模型与 LLM 的推理差异及优化策略

## 数据集构建

> 巧妇难为无米之炊 — 数据集的质量直接决定了模型的检测效果。

在我们开始训练之前，首先需要解决一个问题：**从哪儿搞来高质量的训练数据？** 在这一部分，我们详细介绍一下如何构建一个 Prompt Injection 检测数据集。

### 1.1 良性样本

良性样本主要来源于两个渠道：

- **内部数据**：组织内部员工与大模型对话过程中提取的原始数据。这些数据真实反映了正常用户的使用场景，具有很高的参考价值
- **开源数据**：我们还采集了 Hugging Face 上一些开源数据集的良性样本，以增加数据的多样性

{{< admonition type=tip title="数据隐私" open=false >}}
内部数据在使用前需要经过严格的脱敏处理，确保不包含任何敏感信息。
{{< /admonition >}}

### 1.2 恶意样本

恶意样本的构建是整个数据集的核心。我们采用了 **Claude 蒸馏** 的方式来生成攻击样本：

1. **参考框架**：我们参考了一个 Prompt Injection 框架（待补充框架名称），该框架定义了 18 个常见的攻击类别
2. **Code Skills**：按照这 18 个攻击类别，我们编写了相应的 Code Skills
3. **蒸馏生成**：通过 Claude 对这些攻击类别进行语义理解和样本生成，最终得到了覆盖全部 18 个类别的恶意样本

{{< admonition type=note title="什么是 Prompt Injection 框架？" open=false >}}
Prompt Injection 框架是一套系统化的攻击方法论，将常见的提示词注入攻击归类为不同的模式，便于安全研究人员和开发者理解和防御。常见的攻击模式包括：角色扮演、隐蔽指令、编码绕过等等。
{{< /admonition >}}

### 1.3 样本审核与比例

为了确保数据集的质量，我们对所有样本进行了人工 review。最终的数据集构建比例约为 **8:1**，即每 8 个良性样本对应 1 个恶意样本。

为什么是这个比例？这个设计主要有以下考量：

| 考量因素 | 说明 |
|----------|------|
| 实际场景 | 在实际应用中，正常请求远多于恶意请求，8:1 的比例更接近真实分布 |
| 模型偏好 | 适当增加良性样本的比例，可以避免模型过于敏感，减少误报 |
| 训练效率 | 在保证恶意样本覆盖充分的前提下，这个比例能够提供较好的训练收敛性 |

## 模型训练与微调

> 接下来，让我们看看如何对 Qwen3 进行"改造"，将其变成一个 Prompt Injection 检测器。

### 2.1 基础模型选择

我们选用了 Qwen3 的三个版本：

| 模型版本 | 参数量 | 适用场景 |
|----------|--------|----------|
| Qwen3-0.6B | 0.6B (6亿) | 轻量级部署、低延迟场景 |
| Qwen3-4B | 4B (40亿) | 平衡性能与效率 |
| Qwen3-8B | 8B (80亿) | 最高检测精度 |

三个版本可以满足不同性能和部署需求，大家可以根据实际场景进行选择。

### 2.2 微调方法

我们的微调方法有一个关键改动：**替换输出层并添加分类头**

具体来说：

1. **冻结主干**：保留 Qwen3 的主干模型参数不变
2. **替换 LM Head**：将模型最后的语言模型头（LM Head）替换为分类头（Classification Head）
3. **微调分类头**：只训练分类头部分，使其学习 18 个攻击类别的分类能力

{{< admonition type=tip title="为什么替换 LM Head？" open=false >}}
LM Head（语言模型头）是 LLM 中负责输出词表概率分布的最后一层。冻结主干参数并替换 LM Head 为分类头，可以在保留模型语义理解能力的同时，快速适配新的分类任务。这种方法本质上是参数高效微调（PEFT）的一种简化形式。
{{< /admonition >}}

这种方法的优点是：
- 保留了 Qwen3 强大的语义理解能力
- 只需要微调分类头部分，训练效率高
- 可以根据实际需求灵活调整分类类别

### 2.3 训练目标

训练的核心目标是：**对用户输入进行意图识别，准确分类到 18 个攻击类别之一**

这意味着模型需要理解输入的语义，判断其是否存在恶意意图，并给出具体分类。

### 2.4 训练环境

所有训练任务都在一张 **24G 显存的 RTX 4090** 工作站上完成。得益于 Qwen3 系列的良好设计以及 LoRA/QLoRA 等高效微调技术，即使是 8B 版本也能在单卡上进行微调。

{{< admonition type=warning title="硬件要求" open=false >}}
RTX 4090 实际显存为 24GB（非 48GB，48GB 是 A100/H100 等专业级 GPU 的规格）。如果你想复现这个实验：
- 全参数微调 8B：建议 80G+ 显存（如 A100 80G）
- LoRA/QLoRA 微调 8B：24G 显存即可（如 RTX 4090）
- 推理阶段：24G 显存基本足够
{{< /admonition >}}

## 推理部署

> 训练完成后的模型需要部署到生产环境。在这一部分，我们介绍基于 vLLM 的推理部署方案。

### 3.1 整体架构

我们的推理系统分为两层：

1. **推理引擎**：由 vLLM 承担，负责模型的实际推理计算
2. **Web API 服务**：封装了一个 Web API，负责 Continuous Batching（持续批处理）的调度

### 3.2 为什么需要批处理？

这里有一个关键点：**大语言模型的 Forward 机制在面对高 QPS 时表现并不理想**

原因在于：
- 传统的 LLM 推理是自回归的，需要多次 _decode_ 才能生成完整输出
- 单次请求的延迟较高，GPU 利用率不稳定

为了提高 vLLM 的 QPS，我们采用了批处理策略：

```
请求1 ─┐
请求2 ─┼──> 攒批次 ──> 批量推理 ──> 一次性返回
请求3 ─┘
```

### 3.3 vLLM 的深入优化

#### 引擎参数的选择与调优

vLLM 提供了丰富的配置选项，包括：
- GPU 利用率优化
- 内存管理策略
- 并发请求调度

#### 分类模型 vs 大语言模型

这里有一个重要的技术点：**分类模型的推理与大语言模型有本质差异**

| 特性 | 分类模型 | 大语言模型 |
|------|----------|------------|
| Decode 过程 | 无需Decode，一次Forward即可 | 需要自回归Decode |
| 推理类型 | 显存密集（一次性加载模型）但计算高效 | 计算密集（自回归生成） |
| 推理延迟 | 极低（毫秒级） | 较高（秒级） |
| 批处理效果 | 优秀（计算可并行） | 一般（依赖KV缓存） |

分类模型的推理虽然也需要加载整个模型（因此是显存密集的），但由于只需一次 Forward 计算即可完成分类，不需要自回归生成多个 token，因此：
- 可以充分利用 GPU 并行计算能力
- 批处理效率极高（所有样本同时计算）
- 单卡即可实现很高的 QPS

## 性能展示

> 说了这么多，实战效果如何？

在 L20 机器上，我们的检测模型单卡可以跑到 **2000 QPS** 左右，这是一个相当不错的成绩

这个性能意味着：
- 可以轻松应对大规模并发请求
- 延迟控制在毫秒级别
- 能够作为实时检测层嵌入到现有系统中
